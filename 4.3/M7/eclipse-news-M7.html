JDT
Remove type parameters Ever encountered a case where content assist adds type parameters for a generic type, but you do not want them (example: while invoking a static method)? Now, rather than deleting them individually, you can just delete the '<' and that will remove the entire text up to '>'.
Propose templates and keywords without prefix Content assist now proposes templates and keywords without typing a prefix first:

JUnit test templates The JUnit test templates have been renamed to show the JUnit 4 template before the JUnit 3 template:

JUnit view shows assumption failures The JUnit view now counts @Ignored tests and tests that didn't meet an assumption as 'skipped tests'.
Quick Fix to create 'for' loop variable The new Create loop variable quick fix corrects an incomplete 'for' loop by adding the type of the loop variable. 
	public void foo(ArrayList<? extends HTMLElement> elements) {
		for (e: elements) {
		}
	}
Semicolon can be used as a trigger to select any method Irrespective of the type of method, you can now use ';' as a trigger to select any method during content assist. The ';' will be appended at the end of the selected method.
Appends ';' to void methods during code completion If a void method is selected during content assist, then the Java editor code completion will now automatically append a semicolon at the end of the selected method.
PDE
Default start level settings apply to all plug-ins When editing the Plug-ins tab of a Eclipse Application launch configuration the default start level and default auto start settings chosen at the top of the tab will be used when launching with "All workspace and enabled target plug-ins". Previously this setting would only be used when launching a specific list of plug-ins.

Running from PDE editors remembers previous selection The PDE editors allow applications to be launched from the top toolbar of the Manifest, Plug-in and Product editors. By default PDE only provides one type of launch. If you have additional tooling installed such as RAP Tools, all possible launches will be available in a drop down menu.

This menu now remembers which launch was used most recently and puts it at the top of the list. The launch chosen when the button is pressed will be the most recent launch. The order is saved between workbench restarts.

JUnit Plug-in tests can run on e4 workbench Applications that use the e4 workbench API such as e4 RCP applications can now use JUnit Plug-in Test launch configurations to test their plug-ins. Previously the tests would require the 3.x workbench API from the org.eclipse.ui bundle to hook into the workbench lifecycle.
API Tools has limited support for pre-OSGi Eclipse plug-ins Eclipse plug-ins created before 3.1 did not contain an OSGi bundle manifest. API Tools now can convert the pre-OSGi plug-ins to valid components in a API baseline. This allows analysis, use and freeze scans to process plug-ins that depend on pre-OSGi plug-ins instead of failing to resolve.

However, an OSGi runtime is required to do the conversion. Tasks run using EclipseAntRunner or the API Use Report external tools launch configuration can convert the plug-ins. Tasks run from the command line ant runner will skip pre-OSGi plug-ins.

API Tools allows @noreference javadoc tag on types API Tools now allows the use of the @noreference javadoc tag on types (classes, interfaces, annotations and enums). Placing this tag restricts the API so that any reference to that type or its members will be flagged as invalid API use.

To mark a type that is in an API package as not being public API it should be tagged as @noreference, @noextend and @noinstantiate (or @noimplement). This ensures that no client can access it via valid API and the type could later be removed.

API Tools javadoc tags check parent visibility If a member is not publicly visible (ex: a private method) it is not part of the API and any API Tools javadoc tags on it are invalid. If a member is not visible because a parent type is not visible the javadoc tags will now be marked as invalid.
